# 给 HTML 演示添加语音旁白

当演示需要一段 TTS 配音，并希望它**始终与屏幕上的编排对齐**，而不必每次修改脚本都手动重新对时。本参考覆盖工具链（`generate-tts.py` → `transcribe.py` → `dump-timestamps.py`）、"音频作为主时钟"的播放模式，以及那些只有踩过才会知道的坑。

如果演示只需要画面——直接跳过本文档。`SKILL.md` 中的模式可以独立工作。

## 核心思想：音频是主时钟

朴素做法——先生成音频，然后期望它能和时间线对上——每次修改脚本都会漂移。稳健做法：

1. **把旁白写成纯文本**（每个场景一份文件）
2. **生成 TTS 音频**，通过 OpenAI 的 `gpt-4o-mini-tts` 或阿里 DashScope 的 `qwen3-tts-flash`
3. **用 Whisper 转录**生成的音频，得到词级时间戳
4. **定义 checkpoint 锚点**——一个 `[audioTime, sceneTime]` 数组，把口语短语绑定到场景时间节拍
5. **用音频驱动视觉时间线**：`requestAnimationFrame` 循环读 `audio.currentTime`，在 checkpoint 之间线性插值，seek GSAP 时间线

非均匀节奏自然出现。旁白密集的音频段在场景侧播放更快；安静段则给画面留出呼吸空间。

## 项目布局

```
your-project/
├── demo.html                  # the page
├── audio/
│   ├── scripts/
│   │   ├── intro.txt          # narration source (filename → scene id)
│   │   ├── act-1.txt
│   │   └── outro.txt
│   ├── intro.mp3              # ← generated by generate-tts.py
│   ├── intro.json             # ← generated by transcribe.py
│   ├── act-1.mp3
│   └── act-1.json
└── .env                       # OPENAI_API_KEY=sk-...  or  DASHSCOPE_API_KEY=sk-...
```

## 工作流

### 1. 写脚本

每个 `.txt` 文件是一个场景。文件名（不含 `.txt`）即场景 id，会成为 `<audio>` 元素的 id（例如 `audio-intro`）。

效果不错的结构模板：

| 段落 | 时长 | 用途 |
|---|---|---|
| Intro | 4–10 秒 | "Meet [人物] — [角色 / 背景]. [赌注一句话.]" |
| Action | 25–40 秒 | 讲清楚发生了什么，点名画面正在展示的瞬间 |
| Recap | 6–15 秒 | 总结本场景展示了什么——简洁收尾 |

目标是**每场景约 100–140 词**（默认 TTS 速度下落在 50–65 秒音频）。再长会冲掉画面；再短则结尾留死气。

Recap 是你应当显式点名本场景涉及的架构轴线 / 系统组件的地方。观看者听到 "Skills did X, Memory carried Y, Runtime did Z" 时，视觉上的轴线胶囊正好全部点亮。双重编码让词汇真正落地。

### 2. 生成 TTS 音频

`generate-tts.py` 支持两个后端，通过 `--backend` 选择：

```bash
# OpenAI (default voice: ash)
python3 scripts/generate-tts.py \
  --scripts-dir audio/scripts --out-dir audio \
  --backend openai --voice ash

# Alibaba DashScope / Qwen-TTS (default voice: Cherry)
python3 scripts/generate-tts.py \
  --scripts-dir audio/scripts --out-dir audio \
  --backend qwen --voice Cherry --region cn
```

#### 后端选择

脚本默认是 `--backend auto`：
- 如果环境或 `.env` 中只有 `OPENAI_API_KEY` → 用 OpenAI。
- 如果只有 `DASHSCOPE_API_KEY` → 用 DashScope。
- 两个都有时，脚本交互式提问（非 TTY 上下文则带清晰提示退出，让你显式传 `--backend`）。

不确定用户想要哪个时，**先问**而不是猜。两个后端的声线特性差异显著，且 API key 通常对应不同的计费账号——用户选择是有原因的。便于摸清意图的问题："你在哪个地区？" / "你有 OpenAI 还是 DashScope key，或者都有？" / "声线偏好（偏英语 vs 偏 CJK，中性 vs 温暖）？"

#### OpenAI 声线（`gpt-4o-mini-tts`）

| 声线 | 特性 | 适用 |
|---|---|---|
| `ash` | 自信、清晰、现代 | 技术 / 产品演示（默认首选） |
| `sage` | 沉稳、成熟 | 长篇讲解 |
| `verse` | 富有表现力、叙事感 | 故事模式作品 |
| `echo`、`alloy` | 中性 | 不希望声音带强个性时 |
| `onyx` | 低沉、权威 | 企业 / 严肃 |
| `coral`、`nova`、`shimmer` | 友好、轻盈 | 消费向 / 俏皮 |
| `ballad`、`fable` | 叙事 / 故事书 | 品牌故事作品 |

OpenAI 也接受 `--speed 0.85–1.15` 和 `--instructions "tone description"`。instructions 参数对韵律有显著影响——保持简短具体（"calm technical narrator with deliberate pacing" 优于 "engaging and dynamic voice"）。

#### Qwen / DashScope 声线（`qwen3-tts-flash`）

40+ 个声线，包括流利英语（`Cherry`、`Ethan`、`Serena`、`Ryan`、`Jennifer`）和地方 CJK 声线（`Jada` 上海话、`Dylan` 北京话，还有粤语、四川话、韩语、日语等）。DashScope 自动从文本检测语言；我们通过扫描 CJK 字符启发式地设置 `language_type`。

旁白混合英语和较多中文时，Qwen-TTS 是更好的选择——其双语韵律比 OpenAI 更自然。纯英语场景下 OpenAI 的 `ash` 声线很难被超越。对完整演示下决定之前，先在 2 句话的样本上把两个都试一下。

#### 区域端点（仅 Qwen）

DashScope 使用按区域区分的 API key 和 base URL：

| `--region` | 端点 | 适用场景 |
|---|---|---|
| `cn`（默认） | `https://dashscope.aliyuncs.com` | 中国账号 / 中国 API key |
| `intl` | `https://dashscope-intl.aliyuncs.com` | 新加坡账号 / 国际版 key |
| `us` | `https://dashscope-us.aliyuncs.com` | US（弗吉尼亚）账号 / US key |

**Key 在区域之间不可互换**——用中国 key 去打新加坡端点会返回认证错误。脚本默认 `cn`，但会优先使用环境变量 `DASHSCOPE_REGION`。如果你不知道用户的 key 属于哪个区域，问他——他创建 key 时使用的控制台 URL 是关键线索。

#### 输出格式

两个后端默认产出 `.mp3`。OpenAI 直接返回 mp3；Qwen 返回 wav，脚本通过 ffmpeg 转换（`libmp3lame -qscale:a 2`）。传 `--out-format wav` 可跳过转换，保留 Qwen 原始 WAV（24-bit，画质更高但体积约大 5×）。

#### 重新生成技巧

只重新生成有改动的部分：
```bash
python3 scripts/generate-tts.py --only intro,outro
```

如果音频整体始终偏快/偏慢，用 `--speed 0.95` 或 `--speed 1.05`（OpenAI）重新生成。Qwen-TTS 不暴露 speed 参数——改文本（短句读起来更快）。

### 3. 转录获取时间戳

```bash
python3 scripts/transcribe.py --audio-dir audio
```

在每个 mp3 旁边写出 `{scene}.json`，包含 `words`、`segments`、`text` 和 `duration`。

### 4. 找锚点时间

```bash
python3 scripts/dump-timestamps.py --audio-dir audio
```

它会打印短语级片段及起止时间。每个场景挑 6–12 个锚点短语——口语落在视觉节拍上的瞬间。示例输出：

```
══════ intro (43.70s) ══════
   0.00- 2.68  3 a m pager duty fires
   3.46- 5.44  Payment service latency just tripled
   6.20- 9.72  Ops is already on it Skills pull metrics
  ...
```

### 5. 定义场景 checkpoint

在 HTML 的场景配置中：

```js
scenes.intro = {
  sceneEnd: 30.0,                 // GSAP timeline's built duration
  checkpoints: [
    [0.00,  0.0],                 // intro narration → scene idle
    [7.50,  0.0],                 // hold during context setup
    [8.54,  1.5],                 // first action: "pager fires" → PD card slides in
    [16.54, 4.0],                 // "Skills pull metrics" → triage starts
    [28.62, 12.0],                // "Runtime drafts hypothesis"
    [32.92, 17.0],                // "He taps once" → approval beat
    [43.94, 31.0],                // "Five axes did the work" → closing strip
    [61.51, 34.0],                // end of audio → scene end
  ],
  buildTimeline() { /* paused timeline */ },
};
```

经验法则：
- **第一行**始终 `[0.00, 0.0]`
- **最后一行**把音频结尾映射到场景结尾
- 在 **intro 段**期间，把场景时间锁在第一个动作的前置状态（说"meet so-and-so"时别播视觉）
- 在 **outro / 轴线 recap** 段，缓慢把场景时间走向 `sceneEnd`，让结尾卡片在口头总结期间始终可见
- 目标约 6–12 个锚点；任意两个锚点之间，场景时间随音频时间线性插值

## 接通"音频驱动主时钟"的播放

HTML 中六个部件。

**a.** 每个场景一个 `<audio>` 元素：
```html
<audio id="audio-intro" preload="auto" src="./audio/intro.mp3"></audio>
<audio id="audio-act-1" preload="auto" src="./audio/act-1.mp3"></audio>
```

**b.** Checkpoint 插值：
```js
function audioToScene(audioT, scene) {
  const cps = scene && scene.checkpoints;
  if (!cps || !cps.length) return audioT;
  if (audioT <= cps[0][0]) return cps[0][1];
  for (let i = 0; i < cps.length - 1; i++) {
    const [a0, s0] = cps[i];
    const [a1, s1] = cps[i + 1];
    if (audioT <= a1) {
      const span = a1 - a0;
      const frac = span > 0 ? (audioT - a0) / span : 0;
      return s0 + frac * (s1 - s0);
    }
  }
  return cps[cps.length - 1][1];
}
```

**c.** rAF 驱动循环——音频通过 seek 驱动 GSAP 时间线：
```js
let driving = false;
function driveLoop() {
  if (!driving || !currentAudio || !currentTl) return;
  if (currentAudio.paused) { driving = false; return; }
  const sceneT = audioToScene(currentAudio.currentTime, scenes[activeSceneId]);
  currentTl.time(sceneT);
  requestAnimationFrame(driveLoop);
}
function startDrive() {
  if (driving) return;
  driving = true;
  requestAnimationFrame(driveLoop);
}
function stopDrive() { driving = false; }
```

**d.** GSAP 时间线**以暂停状态构建**（`gsap.timeline({ paused: true })`）——你会用 `currentTl.time()` seek 它，而不是让它自动播放。

**e.** 播控操作音频：
```js
// Play:    audio.play() → startDrive()
// Pause:   audio.pause() → stopDrive()
// Restart: scene.reset(); audio.currentTime = 0; audio.play()
// Scrub:   audio.pause(); audio.currentTime = frac * audio.duration; sync timeline once
// Mute:    audio.muted = true (audio keeps playing silently so scene keeps moving)
```

**f.** 进度条刻度的反向映射——每个节拍标记按其音频时间位置展示，而非场景时间：
```js
function sceneToAudio(sceneT, scene) {
  const cps = scene && scene.checkpoints;
  if (!cps || !cps.length) return sceneT;
  if (sceneT <= cps[0][1]) return cps[0][0];
  for (let i = 0; i < cps.length - 1; i++) {
    const [a0, s0] = cps[i];
    const [a1, s1] = cps[i + 1];
    if (sceneT <= s1) {
      const span = s1 - s0;
      const frac = span > 0 ? (sceneT - s0) / span : 0;
      return a0 + frac * (a1 - a0);
    }
  }
  return cps[cps.length - 1][0];
}
```

## 平滑入场 / 退场

- **场景加载时旁白渐入**：设 `audio.volume = 0`，调用 `audio.play()`，在约 0.4 s 内把音量补到 1
- **暂停/恢复**：暂停时做 0.2 s 短淡出，恢复时做短淡入——避免音频咔嗒声
- **场景切换**：立刻停止前一段音频，替换 `currentAudio`，新音频淡入——中间留点静音也没问题，读起来就是场景分隔
- **静音**：优先 `audio.muted = true`，而不是 pause——让音频继续解码，场景才能保持步调推进

## 音频相关坑点

**自动播放策略**是最大的坑。浏览器在没有先前用户手势的情况下会拒绝 `audio.play()`。新加载页面后的第一个场景可能默默播放。一旦用户在任何地方点过——播放按钮、进度条、人物卡——后续整个会话 `audio.play()` 都可用。**别把视觉门控在音频 promise 上**——即便音频被拦截，时间线也要在 t=0 渲染，由播放按钮起步。

**拖动进度条会重新触发 `.call()` 回调**。GSAP 每次播放头前向越过 `.call()` 都会触发。来回快速拖动会重复触发任何挂在回调里的音效或一次性动画。对演示来说可以接受；如果真的需要单次触发，在回调里用"本次运行已触发"标志关上闸，restart 时清除。

**音频加载延迟**。`preload="auto"` 是提示，不是保证。驱动循环要在 `isFinite(audio.duration)` 时才启动。如果 `audio.duration` 是 NaN，退回到场景自身的 `sceneEnd` 来显示已用时间。

**TTS 跨运行不确定**。即便脚本、声线和速度都一样，OpenAI 和 Qwen-TTS 每次调用产出的时间也会略有差异。重新生成后，始终重新跑 `transcribe.py` 并重新挑 checkpoint 锚点。词级时间戳相对你直觉"感觉对"的位置会漂移约 0.1–0.4 s——用耳朵测，根据需要把锚点微调 ±0.3 s。

**DashScope URL 24 小时过期**，但脚本会立即下载，所以这只在你保存了响应 JSON 并打算稍后下载时才有影响。重新跑脚本即可。

**DashScope 区域 / key 不匹配会返回 401**。用新加坡 key 调 `dashscope.aliyuncs.com`（或反过来）的典型情况是 401 报错 "InvalidApiKey"。始终把 `--region`（或 `DASHSCOPE_REGION`）设成匹配 key 区域的值——别假定 `cn`。

**Whisper 转录有时把专有名词弄串**（品牌名变成发音相似的别的词）。时间戳仍然准确；找锚点时去匹配*实际口语*短语（"OpenClaw" 而不是 "OpenPaw"）。

**逐词 vs 逐短语锚定**。词级太细碎且噪声大。挑*短语*锚点——通常是关键句子的第一个词。每场景超过约 12 个锚点收益递减。

**别混用 `timeScale()` 和音频驱动 seek**。如果时间线带有非 1 的 timeScale，`tl.time()` 仍按时间线时间，但实际节奏会发散——你会得到漂移。始终用 `timeScale = 1` 构建时间线，靠 checkpoint 伸缩来控节奏。

## 声线 / instructions 调优

`gpt-4o-mini-tts` 的 `--instructions` 参数对语气影响显著。常用模式：

- **自信的产品演示**：*"Calm, confident, professional product-demo narration. Measured pacing with natural pauses between sentences."*
- **温暖的讲解**：*"Friendly, warm narration like explaining something to a thoughtful colleague. Slow down on technical terms."*
- **电影感 / 戏剧**：*"Cinematic narrator voice. Build tension through measured pacing. Italicize key phrases with emphasis."*
- **多语言混合**：*"Speak Chinese phrases with natural Mandarin pronunciation; tech terms stay English."*

Qwen-TTS 标准的 `qwen3-tts-flash` 模型不接受自由指令——挑合适的声线，并通过文本本身塑造节奏（短句、逗号节奏）。`qwen3-tts-instruct-flash` 模型接受 `instruct` 字段但较慢，`generate-tts.py` 默认不用；仅当你认为声线特性控制比吞吐更重要时再考虑。

在任一后端上都可快速迭代：`--only sceneName` 单独重新生成一个场景以对比声线，再下定决心。

## 脚本修改后重新生成

```bash
python3 scripts/generate-tts.py --only changed-scene
python3 scripts/transcribe.py --only changed-scene
python3 scripts/dump-timestamps.py --only changed-scene
# Update that scene's `checkpoints` array in demo.html
```
